# Epic 1 Retrospective - Project Foundation & Site Management

## Epic Summary

- Epic: `Epic 1 - Project Foundation & Site Management`
- Stories completed: `4/4`
- Retrospective type: `Full retrospective after epic completion`
- Participants: Project Lead, Scrum Master, Product Owner, Senior Dev, QA, Junior Dev

Epic 1 delivered the core platform baseline: project scaffolding, database schema and migrations, dashboard shell, auth middleware, site submission flow, site list filtering/sorting/pagination, and lifecycle actions (skip/re-analyze/delete) with transition guards.

## What Went Well

- Strong architecture alignment from day one: implementation followed shared patterns from `architecture.md` and `epics.md`.
- Service-layer discipline held: business logic stayed in `src/services/` instead of route handlers.
- API contract consistency was maintained (`{ data }`, `{ data, meta }`, structured `{ error }`).
- UX foundations landed early: app shell, status badges, confidence bar, status tabs, and empty/loading states shipped consistently.
- Team adapted quickly to framework details (Next.js proxy behavior, Prisma adapter setup) and documented key decisions in story notes.

## Challenges and Friction

- Framework shifts created onboarding friction:
  - Next.js proxy behavior and route handler parameter shape required careful handling.
  - Prisma driver-adapter setup and generated-client import paths added non-trivial setup complexity.
- Security ergonomics tradeoff emerged:
  - Dashboard API calls rely on `NEXT_PUBLIC_API_TOKEN` for MVP simplicity.
  - This is practical short-term, but not ideal for long-term hardening.
- Verification depth varied by story:
  - Some manual verification checklist items were marked as not executed in-story.
  - Confidence is high on build/lint integrity, but runtime confidence should be strengthened with repeatable checks.

## Recurring Patterns Observed Across Stories

- Positive pattern: implementation became faster and more consistent after `Story 1.1` established conventions and reusable primitives.
- Positive pattern: query invalidation strategy (`["sites"]`) scaled cleanly across create/update/delete/count workflows.
- Risk pattern: testing strategy remains under-defined at the artifact level (stories emphasize build/lint and manual checks more than automated test coverage).

## Technical Debt and Follow-Ups

- Auth model hardening:
  - Replace public token strategy with a safer same-origin/session approach for dashboard API access.
- Verification maturity:
  - Add integration tests around site lifecycle transitions and API status/response contracts.
  - Add focused UI tests for Sites table actions and filtered empty states.
- Operational confidence:
  - Add repeatable smoke checklist for critical routes before moving deeper into worker-heavy epics.
