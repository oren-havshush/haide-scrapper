# Epic 4 Retrospective - Scraping Engine & Data Collection

## Epic Summary

- Epic: `Epic 4 - Scraping Engine & Data Collection`
- Stories completed: `3/3`
- Retrospective type: `Full retrospective after epic completion`
- Participants: Project Lead, Scrum Master, Product Owner, Senior Dev, QA, Junior Dev, Architect

Epic 4 delivered the first full scrape execution loop: trigger flow from dashboard/extension, worker-driven extraction, normalization and validation, and persisted job outputs with scrape-run metrics. This moved the platform from configuration readiness into actual data generation and quality visibility.

## What Went Well

- Core pipeline handoff is now operational:
  - `4-1` reliably creates `ScrapeRun` + `WorkerJob` and exposes scrape status in the dashboard.
  - `4-2` implements worker scrape execution with single-page and multi-page strategies.
  - `4-3` modularizes normalization/validation and makes quality status queryable.
- Reliability controls were implemented early:
  - 2-minute global timeout and detail-page timeout protections.
  - Failure categorization (`timeout`, `structure_changed`, `empty_results`, `other`) aligned with ops needs.
  - Browser cleanup and transactional writes reduced partial-state risk.
- Architecture stayed coherent under backend-worker complexity:
  - Worker logic remained in `worker/` with direct Prisma access.
  - API routes remained thin and service-oriented on app side.
  - Clear distinction between `WorkerJob` processing state and `ScrapeRun` business outcome.

## Challenges and Friction

- Behavior verification still leans heavily on manual validation:
  - Build/lint/typecheck evidence is strong.
  - Automated integration coverage for real-site scrape variability remains limited.
- Selector contract brittleness persists:
  - Extraction quality still depends on selector compatibility from prior analysis/correction stages.
  - Drift risk remains without formal fixture regression packs across representative site shapes.
- Failure semantics are implemented but operational playbooks are still emerging:
  - Categories exist, but “what action next” guidance for each category is not yet fully standardized in process docs.

## Deep Story Pattern Synthesis

### Common Success Patterns

- Story decomposition was effective: trigger orchestration (`4-1`) before worker runtime (`4-2`) before data-quality layer (`4-3`).
- Focused modules (`scrape.ts`, `normalizer.ts`, `validator.ts`) improved maintainability and future testability.
- Existing conventions (response wrappers, typed payloads, service boundaries) accelerated delivery without architectural churn.

### Recurring Review and Quality Themes

- Positive: Explicit status/accounting fields (`totalJobs`, `validJobs`, `invalidJobs`, validation status) improved observability.
- Risk: Scraping logic remains highly behavior-sensitive and needs stronger automated regression to reduce surprise breakage.
- Risk: Cross-epic contract ownership (extension config -> worker extraction -> dashboard interpretation) still needs tighter formalization.

### Breakthrough Moments

- Replacing the scrape-dispatch placeholder with real worker execution unlocked the product’s central value path.
- Failure categorization became a bridge from technical failure to operational triage.
- Validation-status filtering in Jobs API turned raw scrape output into actionable quality-review data.
